# 0002 — CLI command surface

**Status:** Accepted (framework adoption staged) · 2026-07-02 · **superseded** by [0006](0006-close-style-backlog.md) — commander dropped as unused

## Context

The bin (`src/index.ts`) hand-routes subcommands with `args.includes('setup')` /
`args.includes('skills')`, then falls through to the MCP stdio server. Setup is
spread across three files (`setup.ts`, `auto-setup.ts`, `interactive-setup.ts`)
and uses `prompts` + `isTTY` + `utils/cli-ui.ts` for its interactive flow. There
is no `--help`, and `diagnose` exists only as an npm script, not on the bin.

## Decision

Adopt **commander** as a thin router — parse, route, auto-generated `--help` —
wrapping the **existing** wizards as command actions. The framework must **not**
replace the `prompts`/`isTTY`/`cli-ui` TUI; it fronts it.

Command surface:

| Command            | Behavior                                     |
| ------------------ | -------------------------------------------- |
| `ebay-mcp`         | MCP stdio server (default, no subcommand)    |
| `ebay-mcp setup`   | OAuth/credential wizard (dual-mode)          |
| `ebay-mcp skills`  | Install agent skills (dual-mode)             |
| `ebay-mcp diagnose`| Config/connectivity check (non-interactive)  |
| `--help` / `-h`    | Auto-generated by commander                  |

Every command is **dual-mode**: a bare invocation in a TTY opens the menu; flags
or a non-TTY defer and **never hang**; both routes call the same `run*` function.
Dev-only tooling (`sync`, `build-ui`, `test-endpoints`) stays behind `npm run`,
off the bin.

## Consequences

- This change set **adds the `commander` dependency** and records this surface;
  it does **not** rewire `index.ts` yet — the wiring + collapsing the three setup
  files behind one module is a tracked follow-up, so the existing TUI flow is
  untouched in this pass.
- `getErrorMessage` is the error path for command actions (no bespoke handling).
